## `verification_checks`

Stores individual OCR, face, liveness, duplicate, age-check, and agentic
adjudication results. Agentic recommendations are stored here as a check type
rather than in a dedicated table so reviewer/API detail views can reuse the
existing per-session evidence model, cascade behavior, retry-safe replacement
semantics, and `checks.<type>` serializer contract. The row is advisory unless
a later explicit auto-decision workflow consumes it.

| Column | Type | Constraints/default |
| --- | --- | --- |
| `id` | UUID | Primary key |
| `verification_id` | UUID | Not null, FK to `verification_sessions.id`, `ON DELETE CASCADE` |
| `check_type` | `check_type` | Not null |
| `status` | `check_status` | Not null, default `pending` |
| `score` | FLOAT | Nullable |
| `result_json` | JSONB | Nullable |
| `created_at` | TIMESTAMPTZ | Not null, default `now()` |
| `updated_at` | TIMESTAMPTZ | Not null, default `now()` |

Indexes: `verification_id` and `check_type`.

OCR check rows store a safe `result_json` payload with the legacy
top-level fields (`name`, `dob`, `document_number_hash`) plus a
`document` object containing normalized document fields:
`document_type`, `issuing_country`, `full_name`, `date_of_birth`,
`gender`, `document_number`, `expiry_date`, `issue_date`,
`nationality`, and `address`. Each normalized field stores
`value`, `confidence`, `source`, and `label`. Raw OCR text lines and
full document numbers are not persisted; `document_number.value`
contains the same HMAC hash as `document_number_hash`.

`document_quality` rows store safe quality metadata only:

- `readability`: `good`, `fair`, or `poor`
- `image_quality`: `good`, `fair`, or `poor`
- `missing_regions`: list of missing document regions such as `top_edge`,
  `bottom_edge`, `left_edge`, `right_edge`, or `portrait`
- `suspected_tampering`: boolean
- `retry_recommended`: boolean
- `quality_confidence`: 0.0-1.0 model/heuristic confidence
- `provider`: `heuristic`, `multimodal_llm`, or `unavailable`

Document quality rows must not persist raw document images, local storage
paths, model prompts, API keys, or provider credentials.

`agentic_review` rows store the validated structured verdict and safe
references only:

- `recommended_status`, `confidence`, `reason_codes`, `human_summary`,
  `evidence_references`, and `requires_manual_review`
- `policy_gate` with `should_call_model`, `skip_reason`, `reason_codes`,
  and `terminal_overrides`
- `provider` metadata such as provider/model name, latency, token/cost
  estimates, fallback reason, model-called flag, and output validation status
- `thread_id` for verification-scoped LangGraph replay/resume
- `evaluation` fields comparing deterministic status to agentic recommendation
  for shadow-mode rollout metrics
- nullable `reviewer_feedback`, reserved for whether a human reviewer agreed
  with the agent once the review UI exists

Agentic rows must not persist raw prompts, raw OCR text, full document numbers,
raw storage paths, API keys, webhook secrets, or biometric captures.

### LangGraph Checkpoint Tables

When `AGENTIC_CHECKPOINT_BACKEND=postgres`, LangGraph stores checkpoint rows in
its own PostgreSQL-managed tables. Create or update those tables during
deployment with:

```bash
uv run setup-agentic-checkpointer
```

The application uses a stable
`verification:{verification_id}:agentic_adjudication` thread id so interrupted
human-review runs can be resumed after worker/API restarts. The verification
worker keeps its normal non-blocking shadow/assist path; interrupt/resume is
reserved for explicit reviewer workflows.

| `external_user_id` | VARCHAR(255) | Not null |

| `purpose` | VARCHAR(50) | Not null default `duplicate_detection`; allowed values initially `duplicate_detection`, `ban_match` |
| `retained_after_subject_delete` | BOOLEAN | Not null default `false`; true only for active ban markers |
| `deleted_at` | TIMESTAMPTZ | Nullable soft-delete marker for lifted bans / cleanup |
| `created_at` | TIMESTAMPTZ | Not null, default `now()` |

Indexes: `client_id`, `organization_id`, `workspace_id`, `subject_ban_id`,
`(workspace_id, purpose)`, and partial ivfflat
`ix_face_embeddings_workspace_ban_embedding` for active `ban_match` rows.

Duplicate and ban searches must always include tenant scope (`client_id` today,
`organization_id` + `workspace_id` as the target). Embeddings must never be
compared across clients/workspaces.

Phase 9 finalised the deferred similarity index. The new Alembic revision
`phase9_duplicate_detection` adds:

```sql
CREATE INDEX ix_face_embeddings_client_embedding
  ON face_embeddings
  USING ivfflat (embedding vector_cosine_ops)
  WITH (lists = 100);
```

The index is keyed on `(client_id, embedding)` implicitly via the
`client_id` predicate - all production search queries filter by
`client_id` first (per `API_CONTRACTS.md`), so the planner can use the
tenant partition of the index. The cosine operator class matches the
Phase 7 normalised-embedding similarity function. `lists = 100` is the
recommended starting point for an MVP-scale row count and should be
revisited if the per-tenant table grows past ~100k rows.

## `subject_bans`

Stores client-controlled soft and permanent bans for external users. This is
workspace-scoped because the same organization may run multiple apps with
different abuse policies.

| Column | Type | Constraints/default |
| --- | --- | --- |
| `id` | UUID | Primary key |
| `organization_id` | UUID | Not null, FK to `organizations.id`, `ON DELETE CASCADE` |
| `workspace_id` | UUID | Not null, FK to `workspaces.id`, `ON DELETE CASCADE` |
| `external_user_id` | VARCHAR(255) | Not null |
| `kind` | `subject_ban_kind` | Not null |
| `reason` | TEXT | Nullable |
| `metadata` | JSONB | Not null, default `{}` |
| `ban_expires_at` | TIMESTAMPTZ | Nullable; required for `soft_ban`, null for `permanent_ban` |
| `is_active` | BOOLEAN | Not null, default `true` |
| `created_by_user_id` | UUID | Nullable FK to `users.id`, `ON DELETE SET NULL` |
| `created_by_api_key_id` | UUID | Nullable FK to `api_keys.id`, `ON DELETE SET NULL` |
| `lifted_at` | TIMESTAMPTZ | Nullable |
| `lifted_by_user_id` | UUID | Nullable FK to `users.id`, `ON DELETE SET NULL` |
| `created_at` | TIMESTAMPTZ | Not null, default `now()` |
| `updated_at` | TIMESTAMPTZ | Not null, default `now()` |

Indexes: `organization_id`, `workspace_id`, `external_user_id`,
`(workspace_id, external_user_id)`, and active bans by
`(workspace_id, is_active, ban_expires_at)`.

Rules:

- One active ban per `(workspace_id, external_user_id)`.
- Soft bans are active until `ban_expires_at`; permanent bans are active until
  explicitly lifted.
- Creating/updating a ban deletes verification sessions, files, checks, and
  webhook deliveries for the subject, but preserves at least one minimized
  `face_embeddings` row with `purpose = "ban_match"` when a usable selfie
  embedding exists.
- Lifting a ban soft-deletes or disables retained ban embeddings unless another
  active ban references them.
- Ban rows and ban-match embeddings are never exposed with raw vectors through
  API responses.

## Data Retention

- Verification sessions and uploaded evidence: retained until explicit
  client-requested reset/delete/ban cleanup or a future configurable retention
  policy. There is no automatic 30-day deletion rule for verification session
  data.
- Face embeddings: retained for duplicate detection until client-requested
  reset/full deletion; retained for ban matching only while an active ban
  requires them.
- Audit logs: at least one year; lifecycle mutation tombstones are permanent
  compliance records unless legal policy changes.
- Webhook logs: 90 days.

Retention jobs are application concerns and are not implemented as database
TTLs.

### Existing table changes

- `workflows`: `organization_id` and `workspace_id` are implemented as nullable
  compatibility columns and backfilled for existing client workflows. Workflow
  names remain unique only within a workspace if uniqueness is added later.
- `api_keys`: `organization_id`, `workspace_id`, and `environment` are
  implemented. API keys belong to exactly one workspace once migrated.
- `verification_sessions`: `organization_id`, `workspace_id`, and `api_key_id`
  are implemented as nullable compatibility columns and backfilled for existing
  sessions. `reviewed_by_user_id`, `reviewed_at`, `review_decision`, and
  `review_notes` remain pending.
- `webhook_deliveries`: `organization_id` and `workspace_id` are implemented
  for support and workspace reporting.
- `webhook_endpoints`: workspace-managed endpoint records are implemented for
  list/create management routes. Session dispatch still uses the per-session
  `verification_sessions.callback_url` until default endpoint dispatch is
  explicitly added.
- `audit_logs`: `organization_id` and nullable `workspace_id` are implemented.
  `actor_user_id`, `actor_type`, `entity_type`, `entity_id`, `metadata`,
  `ip_address`, and `user_agent` remain planned enrichment columns.
- `client_credit_accounts`: migrated from `client_id` to `organization_id` primary scoping.
- `credit_reservations` and `credit_ledger_entries`: fully migrated to `organization_id` primary scoping with `workspace_id` attribution.


Tenant-scoped queries must filter by both `organization_id` and `workspace_id`
where the resource is workspace-level. API-key-authenticated external routes
must derive both ids from the API key rather than trusting request payloads.
